# 06-02 — StarDocks in Cloud Mode

## Objective
Add StarDocks to the cloud galaxy so players can buy ship upgrades.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- D1 schema: `sectors` table has no stardock flag
- Local game: StarDocks are in `galaxy.stardocks` array, checked via `D` key
- Upgrades are handled by `POST /api/action/upgrade` (needs creation)

## Tasks
1. Create D1 migration: `ALTER TABLE sectors ADD COLUMN stardock INTEGER DEFAULT 0`
2. Update `scripts/seed.ts` to pick 3–5 sectors as stardocks (one near FedSpace)
3. Regenerate seed SQL and apply to cloud D1
4. Update `buildGalaxyFromCloud()` to populate `galaxy.stardocks` from the flag
5. Create `POST /api/action/upgrade` endpoint:
   - Validate player has enough credits
   - Check upgrade prerequisites (Mk I before Mk II)
   - Update `player_ships.upgrades_json`
   - Deduct credits
6. Wire `D` key in `CloudSectorScreen` to open upgrade menu
7. Display owned upgrades and available upgrades

## Verification
- [ ] `curl /api/galaxy/1/sector?id=X` returns `stardock: 1` for stardock sectors
- [ ] CLI cloud mode: `D` works at stardock sectors
- [ ] Buying an upgrade deducts credits and updates ship stats
- [ ] `curl POST /api/action/upgrade` with valid payload succeeds
- [ ] Invalid upgrade (insufficient credits) returns 403

## Done When
- StarDocks exist in the cloud galaxy
- Players can buy upgrades in cloud mode
- Upgrade persistence works across sessions
- D1 migration is applied safely to existing data

## Notes
- Stardock sectors should be visually distinct (like local mode)
- Re-seeding the galaxy will wipe existing player ships — consider a targeted UPDATE instead
- The upgrade catalog (`UPGRADE_CATALOG` from engine) should be available client-side
