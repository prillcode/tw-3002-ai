# 02-01-PLAN — Data Models & Galaxy Generator

## Objective
Replace the 10-sector hardcoded `mockGalaxy.ts` with a proper galaxy generation engine. Define strict TypeScript types for all game entities and build a procedural galaxy generator that creates diverse, navigable 100-sector galaxies with seeded reproducibility.

This is the foundation: everything else (economy, combat, ships) builds on these types and this generated galaxy.

## Context
- See `../BRIEF.md` for TW-02 scope
- See `../ROADMAP.md` Phases 1 & 2
- Current mock data: `cli/src/data/mockGalaxy.ts` (10 sectors, hardcoded)
- PRD Section 4: `docs/TW3002-PRD.md`

## Tasks

### 1. Define core types (`engine/types.ts`)

Create strict TypeScript interfaces for all game entities:

```typescript
// Galaxy structure
interface GalaxyConfig {
  sectorCount: number;
  seed: number;
  fedSpaceRadius: number;    // sectors around center that are safe
  portDensity: number;       // 0-1, fraction of sectors with ports
  stardockCount: number;
}

interface Galaxy {
  id: string;                // unique galaxy identifier
  seed: number;
  sectors: Map<number, Sector>;
  connections: Connection[];
  fedSpace: number[];        // sector IDs in FedSpace
  stardocks: number[];       // sector IDs with StarDocks
  createdAt: string;
}

// Sectors
interface Sector {
  id: number;
  name: string;
  coords: { x: number; y: number };  // for map rendering
  port?: Port;
  danger: 'safe' | 'caution' | 'dangerous';
  region: string;            // flavor region name
}

// Ports
interface Port {
  name: string;
  class: PortClass;
  trades: PortTrade[];       // what this port buys/sells
  inventory: Record<Commodity, number>;
  restockRate: number;       // units per turn
}

type PortClass = 1 | 2 | 3;  // 1=best, 2=medium, 3=worst

interface PortTrade {
  commodity: Commodity;
  direction: 'buy' | 'sell'; // buy = port buys from player, sell = port sells to player
  basePrice: number;
}

type Commodity = 'ore' | 'organics' | 'equipment';

// Connections
interface Connection {
  from: number;
  to: number;
  type: 'warp' | 'toll';    // toll = costs credits to use
  tollCost?: number;
}
```

**Key decisions baked in:**
- `Port` has explicit `trades[]` instead of a single type — ports in TW2002 trade multiple commodities
- `Connection` supports toll lanes (future: paid express routes)
- `Galaxy.sectors` is a Map for O(1) lookup by ID
- Coordinates stored for map rendering
- Danger derived from distance to FedSpace, not hardcoded

### 2. Build name generator (`engine/galaxy/names.ts`)

Procedural name generation for sectors, ports, and regions:

- Sector names: Combine prefix + suffix pools (e.g. "Kepler" + "Station" = "Kepler Station")
- Port names: Class-based suffixes (Class 1 = "Prime", Class 2 = "Outpost", Class 3 = "Depot")
- Region names: 5-8 named regions (e.g. "The Core", "Outer Reach", "Pirate Alleys")
- Seeded random: Same seed always produces same names
- Pool size: ~30 prefixes, ~20 suffixes per category = 600 unique combinations

### 3. Build connection algorithm (`engine/galaxy/layout.ts`)

Generate a navigable, interesting warp lane network:

- Place sectors in 2D grid with jitter (for visual spread)
- Ensure connectivity (minimum spanning tree first, then add extra lanes)
- Target 2-4 connections per sector (TW2002 average)
- FedSpace sectors get more connections (hub-like)
- Outer sectors get fewer (frontier feel)
- No orphan sectors (every sector reachable)
- Max 1-hop to reach at least 1 port from any sector

Algorithm sketch:
1. Place `sectorCount` sectors on grid with jitter
2. Compute minimum spanning tree (Prim's)
3. Add random extra connections until avg degree ≈ 3
4. Verify all sectors have path to nearest port ≤ 2 hops

### 4. Build galaxy generator (`engine/galaxy/generator.ts`)

Main entry point: `createGalaxy(config: GalaxyConfig): Galaxy`

Orchestration:
1. Initialize seeded RNG from config.seed
2. Place FedSpace (center cluster of ~5-10 sectors)
3. Generate region boundaries
4. Place remaining sectors with coords
5. Run connection algorithm
6. Assign port types based on region and density
7. Place StarDocks (1-2 per galaxy, in safe/central locations)
8. Calculate danger levels (distance from FedSpace)
9. Generate names for everything
10. Initialize port inventories

Default config for new game:
```typescript
const DEFAULT_CONFIG: GalaxyConfig = {
  sectorCount: 100,
  seed: Date.now(),         // random unless specified
  fedSpaceRadius: 2,        // 2 hops from center
  portDensity: 0.4,         // 40% of sectors have ports
  stardockCount: 1,
};
```

### 5. Build economy pricing (`engine/economy/pricing.ts`)

Replace `getMarketData()` from mockGalaxy with a proper pricing engine:

```typescript
interface PriceEngine {
  getPrices(port: Port): PriceQuote[];      // current buy/sell prices
  executeTrade(port: Port, trade: Trade): TradeResult;
  tick(galaxy: Galaxy): void;               // advance economy 1 tick
}

interface PriceQuote {
  commodity: Commodity;
  buyPrice: number;       // player pays to buy from port
  sellPrice: number;      // player receives when selling to port
  available: number;      // port stock
  demand: number;         // 0-100 demand level
}
```

Supply/demand model:
- Base prices: Ore=100, Organics=50, Equipment=200
- Port class affects spread: Class 1 (5%), Class 2 (10%), Class 3 (15%)
- Buying from a port increases local supply → prices drop
- Selling to a port increases local supply → prices drop
- Each port "restocks" slowly per tick
- Regions specialize: ore-rich, agriculture, manufacturing

Export clean API:

### 7. Integrate with existing CLI screens

**Critical: CLI screens must keep working during migration.**

Strategy:
1. Create `cli/src/engine/` with all new code
2. Update screens to import types from `engine/types` instead of `mockGalaxy`
3. Replace `getSector()`, `getNeighbors()` calls with `galaxy.sectors.get()`, `galaxy.connections` queries
4. Replace `getMarketData()` with `engine.economy.getPrices()`
5. Remove `mockGalaxy.ts`
6. Generate galaxy on new game, save to DB alongside player state
7. Load galaxy from DB on continue

**DB schema changes (Phase 6 extension):**
```sql
-- New table: galaxies (one per slot)
CREATE TABLE galaxies (
  slot_id INTEGER PRIMARY KEY,
  galaxy_data TEXT,  -- JSON-serialized Galaxy object
  created_at TEXT,
  updated_at TEXT
);
```

Galaxy is saved as JSON blob — avoids complex relational mapping for 100 sectors.

2. **Names** — Generate 100 sector names, verify no duplicates, verify seeded reproducibility
3. **Connections** — Generate 100-sector galaxy, verify:
   - All 100 sectors reachable via BFS from any sector
   - No sector has > 6 connections
   - No sector has 0 connections
   - Average degree between 2-4
4. **Generator** — `createGalaxy(DEFAULT_CONFIG)` produces valid galaxy:
   - 100 sectors
   - 35-45 ports (40% density)
   - 1 StarDock
   - FedSpace cluster of 5-10 sectors
   - All sectors reachable
5. **Economy** — Get prices from a port, buy ore, verify price increases. Sell ore, verify price decreases.
6. **Integration** — Run `bun run dev`, start new game, navigate sectors, trade at market. Everything works but now with generated galaxy.

## Notes
- **Scope guard:** This plan covers roadmap Phases 1-3 (types + galaxy + economy). Combat, ships, upgrades are Phase 4+ and NOT in scope.
- **Why merge 3 roadmap phases into 1 plan:** They're tightly coupled — types define the galaxy, generator creates it, economy prices its ports. Splitting them creates integration churn.
- **Performance:** 100 sectors with Map lookups is trivially fast. No optimization needed.
- **Seed reproducibility:** Use a simple seeded PRNG (xorshift or mulberry32). Don't over-engineer it.
- **Risk:** CLI screen migration is the riskiest part — existing working game must keep working. Do it incrementally, test after each screen update.
