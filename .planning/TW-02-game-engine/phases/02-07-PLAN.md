# 02-07-PLAN — Game State Manager

## Objective
Replace the ad-hoc state management in `App.tsx` with a centralized, validated game state system. All state mutations go through the engine, all actions are validated before application, and the save/load serialization is robust. This eliminates the current risk of invalid states (e.g., negative credits, cargo over capacity, hull > maxHull).

## Context
- Current state lives scattered in `App.tsx` useState hooks: `shipState`, `currentSectorId`, `upgrades`, `shipClassId`, `galaxy`, etc.
- Updates are direct mutations with no validation layer
- `App.tsx` is ~300 lines of state plumbing
- Engine has types but no centralized state container
- Save/load manually maps between React state and DB rows

## Approach

Create a `GameState` class/value object in the engine that owns ALL mutable game state. The CLI holds a single `GameState` instance and calls engine methods to mutate it. The engine validates every mutation and returns a result (success + new state, or failure + reason).

```
App.tsx ── holds ──► GameState (engine)
                        │
                        ├── validateMove(sectorId) → ok/fail
                        ├── moveTo(sectorId) → new state
                        ├── executeTrade(...) → new state
                        ├── initiateCombat(...) → CombatSession
                        └── serialize() → JSON for save
```

### 2. Create state container (`packages/engine/src/state/GameState.ts`)

```typescript
export class GameStateContainer {
  constructor(private state: GameState) {}

  // Read-only getters
  get galaxy(): Galaxy { return this.state.galaxy; }
  get player(): PlayerShip { return this.state.player; }
  get currentSector(): Sector { return this.state.galaxy.sectors.get(this.state.currentSectorId)!; }
  get effectiveStats(): ShipStats { return computeEffectiveStats(this.state.player.classId, this.state.player.upgrades); }

  // Validated mutations — all return Result<T>
  moveTo(sectorId: number): Result<GameStateContainer>;
  executeTrade(trade: TradeAction): Result<GameStateContainer>;
  applyCombatResult(result: CombatResult): Result<GameStateContainer>;
  purchaseUpgrade(upgradeId: string): Result<GameStateContainer>;
  regenerateShield(): GameStateContainer;
  recordCombat(combat: CombatRecord): GameStateContainer;
  recordTrade(trade: TradeRecord): GameStateContainer;

  // Serialization
  serialize(): string;
  static deserialize(json: string): GameStateContainer;
}
```

**`Result<T>` pattern:**
```typescript
interface Result<T> {
  ok: boolean;
  value?: T;
  error?: string;
}
```

### 3. Create validators (`packages/engine/src/state/validators.ts`)

```typescript
export function canMoveTo(state: GameState, sectorId: number): boolean;
export function canTrade(state: GameState, trade: TradeAction): { ok: boolean; reason?: string };
export function canPurchaseUpgrade(state: GameState, upgradeId: string): { ok: boolean; reason?: string };
export function validateState(state: GameState): string[]; // returns list of invariant violations
```

**State invariants to enforce:**
- `player.credits >= 0`
- `totalCargo <= effectiveStats.maxCargo`
- `player.hull >= 0 && player.hull <= effectiveStats.maxHull`
- `player.shield >= 0 && player.shield <= effectiveStats.maxShield`
- `currentSectorId` exists in galaxy
- `player.turns >= 0`
- Each cargo type `>= 0`

### 5. Refactor App.tsx to use GameStateContainer

**Replace all scattered state with a single state object:**
```typescript
const [game, setGame] = useState<GameStateContainer>(() => 
  createNewGameState(config, 'Unnamed', 'merchant')
);
```

**Replace all manual state updates:**
```typescript
// Before (in SectorScreen callback):
setShipState(prev => ({ ...prev, credits: newCredits, cargo: newCargo }));

// After:
const result = game.executeTrade(trade);
if (result.ok) setGame(result.value);
```

**Delete from App.tsx:**
- `shipState`, `currentSectorId`, `upgrades`, `shipClassId`, `galaxy` → all inside `game`
- `effectiveStats` → `game.effectiveStats`
- `galaxyToJson`/`galaxyFromJson` → `game.serialize()`
- `startNewGame` → `createNewGameState(...)`
- `loadExistingGame` → `createNewGameStateFromSave(...)`

### 6. Update save/load to use serialize/deserialize

**`saveLoad.ts` simplification:**
```typescript
export function saveGame(db: Database, slotId: number, game: GameStateContainer): void {
  const json = game.serialize();
  db.run("INSERT OR REPLACE INTO saves (slot_id, game_json, updated_at) VALUES (?, ?, ?)", 
    [slotId, json, new Date().toISOString()]);
}

export function loadGame(db: Database, slotId: number): GameStateContainer | null {
  const row = db.query("SELECT game_json FROM saves WHERE slot_id = ?").get(slotId);
  if (!row?.game_json) return null;
  return GameStateContainer.deserialize(row.game_json);
}
```

**DB migration:**
```sql
-- Add game_json column for unified serialization
ALTER TABLE saves ADD COLUMN game_json TEXT;

-- Legacy migration: construct game_json from old columns on first load
-- (one-time, then drop old columns in future migration)
```

Keep backward compat: if `game_json` is null, fall back to old column-by-column load.

### 7. Update all screens to consume GameStateContainer

- `SectorScreen`: receives `game: GameStateContainer`, calls `game.moveTo(sectorId)`
- `MarketScreen`: receives `game`, calls `game.executeTrade(trade)`
- `StarDockScreen`: receives `game`, calls `game.purchaseUpgrade(id)`
- `CombatScreen`: receives `game`, calls `game.applyCombatResult(result)`

All screens return `Result` to App.tsx, which either applies or shows error.

### 8. Add turn counter management

- Each jump costs 1 turn (already implemented)
- Track `turnsUsed` in GameState
- Cap at `maxTurns` (from ship class + upgrades)
- If `turns <= 0`: player can still view/navigate but not jump (stuck until turns regen — future feature)

### 9. Add event log

- `CombatRecord` and `TradeRecord` stored in GameState

## Verification

1. **Engine typecheck** — `bun run typecheck` passes
2. **Validator tests** — `canTrade` rejects cargo overflow, `canMoveTo` rejects disconnected sectors
3. **Invariant enforcement** — Manually set negative credits, `validateState` catches it
4. **Serialization round-trip** — Create game → serialize → deserialize → compare equal
5. **Save/load round-trip** — New game → trade → save → load → same state
6. **Legacy migration** — Old-format save loads correctly, then re-saves in new format
7. **App.tsx shrink** — Line count reduced by 50%+ (was ~300, target <150)
8. **CLI typecheck + build** — clean

## Done When
- Single `GameStateContainer` holds all mutable game state
- All state mutations validated by engine before application
- Save/load uses unified JSON serialization
- Legacy save format auto-migrates
- App.tsx is lean state router, not state owner
- Invalid states are impossible to create through normal gameplay
- TypeScript clean, binary builds

## Notes
- **Breaking change:** This refactors the core state model. Test thoroughly after.
- **Performance:** 100-sector galaxy serializes to ~50KB JSON. SQLite handles this trivially.
- **Future-proofing:** `game_json` column makes future schema changes easier — just evolve the GameState shape.
- **Event log:** Don't build a full UI for it in this phase. Just store the data. A log screen can come later.
