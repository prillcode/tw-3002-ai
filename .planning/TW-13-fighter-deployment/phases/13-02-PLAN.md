# 13-02 — Sector Entry Fighter Encounters

## Objective
When a player attempts to enter a sector with hostile deployed fighters, resolve the fighter encounter before normal sector actions.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Phase 1 is complete (`06-01-SUMMARY.md`): buy/deploy/recall + sector fighter ownership exists
- Existing movement endpoint: `POST /api/player/ship/move`
- Existing defeat pipeline: `resolveDefeat()` in `cloud/src/routes/combat.ts`
- No corp mechanics (individual ownership only)

## Design Decisions
1. **Movement remains the entry point** (`/api/player/ship/move`) to avoid client divergence.
2. If target sector has hostile fighters requiring a player decision:
   - API returns `encounterRequired: true` and does **not** complete move yet.
3. Client resolves via new endpoint:
   - `POST /api/fighters/encounter/resolve`
4. Offensive mode is auto-resolved server-side (no player choice needed).
5. Encounter outcomes are explicit and machine-readable so UI can narrate cleanly.

## API Changes

### Update `POST /api/player/ship/move`
Current behavior: validates connection, decrements turns, moves ship.

New behavior:
- Check target sector for hostile fighters before committing move.
- If no hostile fighters: unchanged behavior.
- If hostile fighters:
  - Defensive / Tolled mode present: return `409` with encounter payload (no move yet)
  - Offensive-only hostile fighters: auto-resolve and return outcome

**Encounter payload shape (example):**
```json
{
  "encounterRequired": true,
  "targetSector": 42,
  "fighters": [
    { "ownerId": 17, "ownerName": "Nightwing", "count": 1200, "mode": "defensive" }
  ],
  "options": ["attack", "retreat", "surrender", "pay_toll"],
  "tollCredits": 6000
}
```

### New `POST /api/fighters/encounter/resolve`
**Body:** `{ galaxyId, targetSectorId, action }`

`action` values:
- `attack`
- `retreat`
- `surrender`
- `pay_toll` (only valid if tolled fighters present and player can pay)

**Response includes:**
- `outcome` (`entered` | `retreated` | `surrendered` | `destroyed`)
- fighter losses by side
- credits paid/looted
- updated player ship snapshot (or respawn info if destroyed)
- narrative string + event list for UI

## Rules to Implement

### Defensive fighters (1:1)
- Prompt player: attack / retreat / surrender
- **Attack:** resolve losses at 1:1
- **Retreat:** stay in current sector, consume 1 turn
- **Surrender:** eject to nearest FedSpace OR apply 10% credit penalty (choose one and keep consistent)

### Offensive fighters (2:1 pressure)
- Auto-attack behavior
- Defender commits attack strength multiplier consistent with roadmap (`1.25×` force requirement rule)
- If player survives threshold, they may enter sector
- If not, trigger `resolveDefeat()`

### Tolled fighters
- Player can pay `5 credits × hostile fighter count` to pass
- If cannot/will not pay, resolve as 1:1 attack path

### General
- Never allow encounter bypass via direct API path
- Deployed fighter ownership remains per-player
- Encounter resolution always updates `sector_fighters` counts and `player_ships` state atomically

## Web Client Changes
