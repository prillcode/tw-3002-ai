# 13-03 — Ordered Combat Sequence + Ship-to-Ship Resolution

## Objective
Implement TW2002-authentic sector entry sequencing and automatic ship-to-ship combat when fighter mediation does not end the encounter.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- Depends on completion of `13-02-PLAN.md` (fighter entry encounter flow)
- Existing defeat consequences already centralized in `resolveDefeat()`
- Mines and Q-cannons are not implemented yet; sequence must still be scaffolded in correct order

## Required Combat Order
On sector entry, processing order must be:
1. Nav hazard check (if present; no-op for now)
2. Limpet attachment (Phase 13-04)
3. Armid detonation (Phase 13-04)
4. Q-cannon fire (Phase 13-05 / TW-14 dependency)
5. Fighter encounter (Phase 13-02)
6. Ship-to-ship combat if players co-occupy and survive

For this phase, implement items 5-6 fully and scaffold placeholders for 1-4 with explicit operation-log markers (`skipped_not_implemented`).

## Design Decisions
1. **Create a single entry pipeline helper** (e.g. `resolveSectorEntryPipeline`) to prevent logic spread across routes.
2. **Ship-to-ship is automatic** when conditions are met:
   - two or more player ships in the same sector after fighter resolution
   - no duel/challenge button
3. Use existing defeat pipeline for final consequences (loot/respawn/news consistency).
4. Return structured operation log for UI and future replay/debug.

## Data / API Changes

### Movement response extension
Enhance move and encounter-resolution responses with:
```json
{
  "operations": [
    { "step": "nav_hazard", "status": "skipped_not_implemented" },
    { "step": "fighters", "status": "resolved", "details": { "hostileDestroyed": 320 } },
    { "step": "ship_to_ship", "status": "resolved", "details": { "winner": "player:17" } }
  ]
}
```

### Sector occupant query support
Add helper query to load active player ships in a sector excluding self:
- fields: `player_id`, `ship_name`, `hull`, `shield`, `class_id`, `fighters`, `turns`, `created_at`
- sorted by sign-on order proxy (stable deterministic sort):
  - `players.created_at ASC`, then `player_id ASC`

> If true sign-on ordering is unavailable, use deterministic proxy now and document for future enhancement.

## Ship-to-Ship Rules (Phase 13-03 scope)

### Trigger
After fighter phase, if intruder enters and at least one hostile player ship is present in sector.

### Resolution order
1. Empty ships attack first (scaffold hook; if no concept yet, log as skipped)
2. Player ships attack in deterministic sign-on order
3. Sector fighters post-ship sweep (for now, log only if already consumed in phase 13-02)

### Combat model (v1)
- Use simple stat-based exchanges with tunable constants
- One round minimum; optional capped rounds (e.g. max 3) to avoid long API cycles
- If a ship hits 0 hull, call `resolveDefeat()`
- Survivors remain in sector
