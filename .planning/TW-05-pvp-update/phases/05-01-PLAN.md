# 05-01 — PvP Combat Foundation

## Objective
Players can attack each other in shared cloud galaxies.

## Context
- See `../BRIEF.md`
- See `../ROADMAP.md`
- `pvp_kills` table exists in D1 schema but is unused
- Combat resolver already works for NPC combat — reuse it

## Tasks
1. Create `POST /api/action/pvp/attack` endpoint:
   - Validate attacker is authenticated
   - Validate target player is in same sector
   - Check protections: FedSpace, new player (<10k net worth or <24h old), self-attack
   - Return target player data (name, hull, shield, ship class)
2. Create `POST /api/action/pvp/resolve` endpoint:
   - Run combat resolver (same as NPC combat)
   - Calculate loot: 50% credits + 10% cargo (random commodity)
   - Update attacker: +loot, +kills
   - Update victim: respawn in FedSpace, 90% credits, cargo lost, +deaths
   - Insert news item: "Attacker destroyed Victim in Sector X"
   - Insert into `pvp_kills` table
3. Create `GET /api/action/pvp/status` endpoint:
   - Return whether PvP is possible in current sector
   - List other players in same sector
4. Add PvP option to CloudSectorScreen combat flow:
   - If other players present, show "Attack Player" option
   - Select target player, then action (attack/flee/bribe)
5. Protection systems:
   - New player invulnerability check
   - FedSpace block
   - Self-attack prevention

## Verification
- [ ] Two test accounts in same sector
- [ ] Account A attacks Account B
- [ ] Combat resolves with damage numbers
- [ ] Winner gets loot, loser respawns
- [ ] News item generated
- [ ] FedSpace blocks PvP
- [ ] New player (<10k) is invulnerable

## Done When
- Players can attack each other
- Combat is fair (same resolver)
- Protections work
- News ticker shows PvP events

## Notes
- Loot should be significant but not devastating — 50% credits is steep but matches TW2002
- Consider adding a "warning" before attacking ("This player is new — are you sure?")
- Offline attacks: queue them and process on victim's next login
